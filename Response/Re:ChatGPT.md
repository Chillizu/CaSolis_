# 1. 关于 RND 指标

你问：

> 用 RND prediction error 的下降速率作为探索效果指标合理吗？

我的答案：

**不够。**

原因很简单：

RND 只衡量：

> 「模型有没有见过这个状态」

不衡量：

> 「这个状态有没有价值」

---

举个例子：

Agent 发现：

```bash
ls
pwd
whoami
date
hostname
uname
```

每一个都可能产生新状态。

RND 开心：

```text
wow!
新状态！
新状态！
新状态！
```

但实际上：

```text
没有任何能力增长
```

---

这是所有 Curiosity RL 的经典问题：

> Novelty ≠ Utility

新颖不等于有用。

---

我建议至少同时监控：

## Novelty

RND Error

```text
越高越新
```

---

## Utility

能力增长

例如：

Agent 当前已掌握命令集合：

```text
ls
pwd
cat
```

发现：

```text
find
grep
```

之后 benchmark 成功率：

```text
50% → 70%
```

那么：

```text
发现有价值
```

---

## Discovery Efficiency

我更喜欢这个指标：

```text
有价值发现数
──────────────
总探索步数
```

例如：

```text
1000次探索

发现：

grep
find
sort

共3个有效能力
```

那么：

```text
3 / 1000
```

---

因为未来你一定会遇到：

```text
RND 很高

能力没增长
```

这种情况。

---

这些算。


不算。

---

否则会被参数组合刷分。

# 5. 一个你还没意识到的问题

这是我认为最大的研究风险。

---

你现在隐含假设：

```text
发现新能力
=
能力提升
```

但现实往往不是。

---

Agent 很可能变成：

```text
技能库越来越大
```

例如：

```text
grep

find

sed

awk

xargs

sort

uniq

cut

tr
```

---

最后：

任务成功率：

```text
65%

→

66%
```

---

为什么？

因为瓶颈根本不是技能。

而是：

```text
规划能力
```

---

这其实是 ReAct 系 Agent 最经典的问题。

不是：

```text
不会工具
```

而是：

```text
不会决定什么时候用工具
```

---

所以我建议你新增一个实验。

---

# Skill Injection Test

给 Agent 人工塞技能。

例如：

```text
grep

find

sed

awk
```

全部直接放进去。

---

然后测：

```text
成功率提升多少
```

---

如果：

```text
提升巨大
```

说明：

```text
缺技能
```

---

如果：

```text
提升很小
```

说明：

```text
缺规划
```

---

这个实验价值极高。

因为它决定：

```text
下一阶段投资源方向
```
