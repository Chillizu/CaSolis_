# 大小模型协作方案调研

## 现状：静态蒸馏（我们现在做的）

```
Gemma4 ──生成数据──→ 小模型训练 ──→ 独立运行
  一次性的          单向学习         之后不再用Gemma4
```

## 更先进的方案

### 方案 A：动态求助（SWE-Protégé 模式）⭐ 最适合我们

```
小模型 一直在跑 ─┬─ 正常: 自己决策 ─── 继续
                 └─ 卡住/困惑: ─→ Gemma4
                    (perplexity高/   │
                     死循环检测)     ↓
                                    提示: "我看到了X，下一步怎么做?"
                                    Gemma4: "试试Y命令"
```

**为什么适合我们：**
- 小模型永远是决策者（保持自主性）
- 只在真正需要时才问 Gemma4（99% 时间自己跑）
- 每次求助后，小模型可以学习（从自己的经验+Gemma4 的指导）

### 方案 B：风险路由（R2V-Agent）

```
小模型 每一步评估 "风险" ─┬─ 低风险: 自己决定
                          └─ 高风险: 交给 Gemma4
```

**区别：** 比 A 更保守，需要训练一个"风险评估器"，复杂一些。

### 方案 C：自我提升（STaSC / SePT）

```
小模型 生成多条候选路径 ─→ 选最好的 ─→ 用自己的好输出训练自己
```

**终极目标，但现在还有点早。** 我们的模型输出质量还不够当自己的老师。

### 方案 D：交互式GRPO（Look Before You Leap）

```
小模型 探索 → 奖励(覆盖度+完成度) → GRPO训练 → 更聪明
```

**太复杂。** GRPO 需要大量采样，我们 CPU 扛不住。

---

## 推荐的路线图

| 阶段 | 做什么 | 时间 |
|------|--------|------|
| **1. 现在** | Gemma4 生成真·思考链，小模型训练学格式+内容 | 立即 |
| **2. 下一步** | 小模型自主跑，卡住时问 Gemma4（动态求助） | 下一轮 |
| **3. 最终** | 小模型自举，不再需要 Gemma4 | 长远 |

## 数据流对比

### 现有（模板）：
```
[THOUGHT] 看看当前目录
[CMD] ls
[OBS] bin dev home ...
思考是模板填空，没有真正的推理
```

### Gemma4 方案：
```
[THOUGHT] 我看到了很多目录和文件。让我检查/etc里有什么有趣的东西...
[CMD] ls /etc
[OBS] adduser.conf apt bash.bashrc ...
[THOUGHT] 有很多配置文件。我想看看网络配置，先检查一下hosts文件
[CMD] cat /etc/hostname
[OBS] 46ef04fdc6b0
[THOUGHT] 这是一个容器主机名，不是真正的网络配置。再看看其他配置文件
```
思考是真实的、上下文相关的、有逻辑链的。
